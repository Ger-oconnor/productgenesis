Engineering & Development

Maturity Framework

Where does your function stand?

Four stages of AI maturity for engineering leaders. The distinction that matters most: evaluation infrastructure and observability are not the same thing, and most teams have only built one.

1
Aware

Your engineers use AI coding assistants — GitHub Copilot, Claude Code, Cursor, or similar. Code is produced faster. Individual productivity has improved. But the development function's operating model hasn't changed: tickets still flow the same way, review processes are the same, and the concept of "a developer's job" is unchanged. AI is a tool engineers use, not a system the team operates.

Tell: "Our engineers use AI coding assistants. Velocity is up."

2
Piloting

You've shipped AI features to production. Some agents are running. Quality is the primary concern — you're seeing failure modes that didn't exist in demos. You may have basic monitoring in place but lack formal evaluation infrastructure. The engineering challenge has shifted from "can we build this?" to "how do we know it's working correctly?"

Tell: "We've shipped AI features. Quality and reliability are our current blockers."

3
Embedding

You have both evaluation infrastructure (offline testing, benchmark suites, red-teaming) AND production observability (tool call success rates, reasoning quality signals, latency distributions, quality drift monitoring). Every agent in production has an owner, a cost baseline, and a defined failure mode. Red-teaming has a dedicated owner and a scheduled cadence before each release.

Tell: "Every agent has eval coverage and production monitoring. Red-teaming is in our release process."

4
Leading

Your development function is AI-native. Developers are System Architects — they describe goals, review outputs, and intervene when agents fail silently. Managed agent infrastructure (Google Antigravity, Claude Managed Agents, OpenAI Codex on-prem) has replaced significant portions of the traditional code-edit loop. The skills you're hiring for and developing have shifted from implementation to system design, prompt engineering, agent evaluation, and failure mode analysis.

Tell: "Developers architect agent systems. Agents do the implementation. We hire for judgment."

Frontier Signals

Where best practice is moving

Synthesised from the most recent and impactful content in the Development category.

The hardest engineering problem in agentic development just became someone else's problem.

Google's Managed Agents API provisions a fully operational sandboxed Linux agent — isolated execution environment, persistent state, tool execution, security controls — with a single API call. Anthropic added MCP tunnels (agents reaching private networks without internet exposure) and self-hosted sandboxes (tool execution inside customer infrastructure). The data sovereignty and compliance barriers that blocked regulated industries from deploying production agents have been addressed at the platform level. The cost-benefit calculation for moving from prototyped agents to production-ready agentic workflows has changed materially.

Source: Google  ·  9to5Mac / Anthropic
Eval tells you if an agent can work. Observability tells you if it is. Most teams have one.

LangChain's survey of 1,300+ professionals found 89% of teams have observability for tracing agent behaviour — but only 52% run offline testing and 37% have online monitoring. The failure mode this creates is specific: teams can see what agents are doing, but can't systematically measure whether they're doing it correctly. LLM systems don't fail with errors — they degrade gradually, in ways that don't show up in standard uptime metrics. Both layers are required, and they address different failure modes.

Source: JetBrains PyCharm Blog  ·  LangChain
Red-teaming is moving from engineering nice-to-have to compliance requirement.

OpenAI's acquisition of Promptfoo — the most widely adopted open-source LLM red-teaming framework — is a governance signal, not a developer tooling story. When the company that builds the model acquires the leading tool for testing that model's safety, evaluation is no longer peripheral to model deployment. Development teams should treat this as a signal to build evaluation infrastructure now, before it is required externally. Teams that build this practice before the mandate have an advantage when the requirement arrives.

Source: DEV Community
The developer's job has changed. The teams that acknowledge it first will hire for the right skills.

The creator of Claude Code hasn't written a line of code in 2026 — shipping dozens of pull requests per day from his phone. Google Antigravity 2.0 ships as an agent orchestration platform, not a coding assistant. The dev role is shifting from primary code author to System Architect and Prompt Director. The teams that invest in system design, prompt engineering, agent evaluation, and failure mode analysis now will have a meaningful advantage over teams that spend 2026 optimising code-generation prompts.

Source: DEV Community  ·  Sequoia AI Ascent 2026
Priority Actions

Your next move

Locate your maturity stage above, then focus on the actions that move you to the next one.

Stage 1 → 2  ·  Aware to Piloting
  • Establish team norms for AI coding tools: what's in scope, what requires review, what's off-limits for sensitive code paths. Unmanaged AI tool use is the first risk to address before expanding.
  • Ship one AI feature to production with a clear failure mode definition. Know in advance what "broken" looks like — because AI systems don't throw errors when they degrade.
  • Measure your current cycle times. You need a baseline before you can evaluate the change that agentic workflows create.
Stage 2 → 3  ·  Piloting to Embedding
  • Build eval infrastructure for every agent before it ships: a benchmark suite of representative failure cases, and a defined quality threshold. If you don't have both, the agent isn't ready for production.
  • Add production observability tuned for AI: tool call success rates, reasoning quality signals, latency distributions, and quality drift — not just uptime. Standard application monitoring is insufficient.
  • Assign a dedicated owner for red-teaming. Schedule a session before every agent release. Build it into the release process, not as an optional hardening step.
Stage 3 → 4  ·  Embedding to Leading
  • Evaluate managed agent infrastructure: Google Managed Agents, Claude Managed Agents, or OpenAI Codex on-prem. Assess which workflows justify the managed sandbox trade-off (reduced observability for managed execution environment).
  • Audit your hiring profile. Are you hiring for implementation skills or for system design, prompt engineering, and agent failure analysis? The skills that matter in 2026 are shifting faster than most job descriptions.
  • Define what "developer as System Architect" means in your team's daily workflow. Which engineers are leading agents? Which are still editing files? That ratio is the maturity metric.
